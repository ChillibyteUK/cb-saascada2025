How It Works

From static schedule to
agreed live structure

SaaScada’s CQRS architecture separates the command engine from the read model, so simulations run against a dedicated projections engine — never against the live ledger. That separation is what makes safe, audit-proof modelling possible at scale.

Step 01

When lending doesn’t fit a template

Traditional core banking systems generate loan schedules from rigid, fixed configurations. Bespoke deal structures — negotiated rates, variable repayment windows, unusual term lengths — don’t fit the template. Teams end up in spreadsheets to model options, creating version control and audit problems before a single transaction has posted.

Why this matters
Static schedules are derived from product settings at origination. Any post-origination negotiation requires manual override — outside the system, outside the audit trail.
Step 02

Snapshot

Open the Testing Toolkit on any non-production account and select Roll Forward Dates. Choose any future date — the account will behave as if that date is today. The snapshot runs entirely against a projection of the account state; no live data is touched at any point.

How to access it
The Testing Toolkit is available in the SaaScada Portal on non-production environments. Open any account, launch the toolkit, and select “Roll Forward Dates” to begin.
Step 03

Model

Once Testing Mode is active, SaaScada simulates every transaction from the real date through to your selected test date — Interest Accrual entries, Interest Charged transactions, and their cascading effect on the account ledger and related balances. Every event is visible in the Transactions tab before you commit to anything.

What gets simulated
Interest Accruals, Interest Charged transactions, and all resulting ledger movements — calculated accurately across the full date range from today to the selected test date.
Step 04

Visualise lifecycle impact

The Account Summary tab displays the full Account Timeline — showing every relevant date window (Deposit, Funding and Maturity) and where the account sits within that timeline under the simulated date. Changes to funding windows, maturity dates and projected balances are immediately visible without running a single live process.

Deposit accounts too
Roll Forward works equally for Fixed Term Deposit accounts — the account moves through Deposit, Funding and Maturity Windows exactly as it would on the selected date.
Step 05

Activate

When the simulated schedule is agreed, it becomes the live future of the loan. Because SaaScada is built on Event Sourcing, every hypothetical event is fully replayable — you can see exactly how the projected balance diverges from current reality at any point in the timeline. The full simulation run forms a complete, immutable audit trail with no manual rekeying required.

Built-in auditability
The Command side uses Event Sourcing — inject a hypothetical event such as a missed payment and replay the loan’s history to see exactly how the projected balance diverges from current reality.
The limitation of static schedules
Derived schedules leave little room for negotiated flexibility.
